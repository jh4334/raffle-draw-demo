# 공공기관 공식 번호 추첨 프로그램 5회 디벨롭 계획

> Claude Code 세션 한도 때문에 Hermes가 PM 계획을 직접 작성함.

## 목표
공공기관의 엄정한 추첨 자리에서 모바일/태블릿/노트북으로 바로 사용할 수 있는 번호 추첨 프로그램으로 개선한다. 추첨 대상은 기본 `1-30`번이며, 진행자가 현장에서 신뢰감을 주고 실수 없이 사용할 수 있어야 한다.

## Round 1 — 번호 추첨 도메인 전환
- 이름 입력 방식에서 `1-30` 번호 풀 기반으로 전환한다.
- 기본 시작 번호 `1`, 끝 번호 `30`, 당첨 수 `1` 제공.
- 번호 범위와 당첨 수 검증.
- 수용기준:
  - 1-30 범위에서만 당첨 번호가 나온다.
  - 당첨 수가 범위 개수보다 크면 `RangeError`.
  - 당첨 번호 간 중복이 없다.

## Round 2 — 공정성/엄정성 UX
- “추첨 대기 → 추첨 중 → 결과 확정” 단계 표시.
- 공식 진행용 문구 추가: 기관명, 행사명, 추첨 일시, 추첨 대상 범위.
- 결과 확정 후 감사 로그에 시각, 범위, 당첨 수, 당첨 번호 기록.
- 수용기준:
  - 추첨 전 조건 요약이 보인다.
  - 결과 확정 시 로그 항목이 추가된다.
  - 로그는 최신순/누적 확인 가능하다.

## Round 3 — 현장 진행자 안전장치
- 재추첨 버튼과 전체 초기화 버튼 분리.
- 재추첨 전 “기존 결과가 감사 로그에 남는다” 안내.
- 결과 복사 버튼 제공.
- 수용기준:
  - 재추첨해도 이전 결과 로그가 사라지지 않는다.
  - 결과 복사 텍스트에 행사명, 범위, 당첨 번호, 시각이 포함된다.

## Round 4 — 모바일/큰 화면 UI 개선
- 모바일에서 버튼/숫자/결과가 크게 보이도록 레이아웃 개선.
- 엄정한 공공기관 분위기에 맞게 IBM Carbon 스타일 적용: 흰색/회색/파랑, 각진 버튼, 명확한 타이포그래피.
- 결과 번호는 큰 숫자 카드로 표시.
- 수용기준:
  - 390px 모바일 폭에서도 입력/버튼/결과가 잘리지 않는다.
  - 주요 버튼 터치 높이가 48px 이상이다.

## Round 5 — 검증/배포/사용 안내
- 단위 테스트와 DOM 스모크 테스트 보강.
- README에 모바일 테스트 링크와 사용 순서 작성.
- GitHub Pages 배포 후 공개 URL 확인.
- 수용기준:
  - `npm test` 전체 통과.
  - DOM 스모크 통과.
  - GitHub Pages URL에서 HTML 응답 200 확인.
