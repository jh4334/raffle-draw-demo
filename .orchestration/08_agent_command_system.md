## 제품 방향
공공기관의 엄정한 추첨 행사에서 사회자/담당자가 모바일 또는 태블릿으로 화면을 보여주며 사용할 수 있는 “공식 번호 추첨 콘솔”.

## 디자인 방향
- IBM Carbon 스타일은 너무 딱딱하고 구려 보일 수 있으므로 폐기.
- Apple/Stripe 계열의 고급스러운 흰색·남색·블루 UI.
- 느낌: “공공기관 공식성 + 금융권 수준의 신뢰감 + 행사 진행용 큰 화면”.
- 모바일 우선. 세로 화면에서 바로 조작 가능.
- 결과 번호는 행사장 뒤에서도 보이도록 크고 선명하게.
- 추첨 전/중/후 상태가 드라마틱하지만 과하지 않게 보여야 함.

## 핵심 UX 시나리오
1. 담당자가 모바일로 접속한다.
2. 기관명/행사명을 확인한다.
3. 기본 1-30 범위와 당첨 수를 확인한다.
4. 큰 CTA 버튼 `공식 추첨 시작`을 누른다.
5. 짧은 긴장감 있는 상태 변화 후 결과가 큰 번호 카드로 확정된다.
6. 감사 로그에 자동 기록된다.
7. 필요 시 결과 문구를 복사한다.
8. 재추첨은 명확히 구분되어야 하며 이전 기록을 삭제하지 않는다.

## 반드시 개선할 것
- 첫 화면에서 “뭘 누르면 되는지” 즉시 보여야 한다.
- 입력 폼이 너무 앞에 나서면 안 된다. 기본값은 접힌 고급 설정처럼 보이게 한다.
- `공식 추첨 시작` 버튼을 최상위 CTA로 만든다.
- 결과 영역을 화면의 주인공으로 만든다.
- 감사 로그는 신뢰감을 주되 복잡하지 않게.
- 모바일에서 스크롤해도 주요 액션이 찾기 쉬워야 한다.
- 버튼 문구를 행사 진행자 말투로 다듬는다.

## 금지사항
- 촌스러운 그라데이션 남발 금지.
- 행정시스템처럼 회색 박스만 쌓는 디자인 금지.
- 모든 요소를 같은 시각적 무게로 배치 금지.
- 테스트 삭제 금지.
- 추첨 로직의 공정성 훼손 금지.
- 외부 빌드 도구/Vite/React 도입 금지. 현재 정적 HTML/CSS/JS 유지.

## 품질 기준
- 모바일 390px 폭에서 CTA/결과/로그가 잘리지 않는다.
- `공식 추첨 시작` 클릭 후 결과가 표시된다.
- 재추첨 클릭 후 감사 로그가 2개 이상 누적된다.
- 결과 복사 기능이 유지된다.
- `npm test` 통과.
- DOM 스모크 테스트 통과.

## Codex 실행 명령 핵심
너는 프리미엄 공공기관 UX를 구현하는 시니어 프론트엔드 엔지니어다. 위 문서를 읽고, 현재 조잡한 UI를 공식 행사에 어울리는 모바일 우선 추첨 콘솔로 전면 개선하라. 단, 정적 HTML/CSS/JS 구조는 유지하고 테스트는 삭제하지 말라.
